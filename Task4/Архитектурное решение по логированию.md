# Архитектурное решение по логированию

## Сбор логов INFO
[jewerly_c4_model.drawio](./jewerly_c4_model.drawio)

Собираем логи из:
- MES API
- CRM API
- Shop API

Собираем следующие данные:
- При создании заказа
  - Номер заказа
  - Идентификатор покупателя
  - Время
- При изменении статуса заказа
  - Идентификатор заказа
  - Время
  - Новый статус
  - При наличии - идентификтаор пользователя, изменившего статус заказа 
- При начале рассчёта стоимости изделия
  - Идентификатор заказа
  - Время
  - Индентификатор файла модели

Другие уровни логирования будут использоваться:
 - ERROR - для логирования ошибок, зафиксированных в системе
 - WARN - для логирования нетипичных ситуаций. Например, переотправка сообщений или запроса
 - DEBUG - для отладочных сообщений, используемых разработкой. Включается при наличии необходимости, в остальное время выключены

## Мотивация
 - Увеличить степень удовлетворения клиентов, обнаружив и исправив места зависания заказа
 - Уменьшить стоимость исправления ошибок путём сокращения трудозатрат работников
 - Уменьшить время обнаружения ошибок в системе путём настройки анализа логов и алёртов

В первую очередь логирование и трейсинг необходимы для систем CRM и MES, т.к. только в этих системах происходит обработка и B2C заказов, и B2B заказов. Нахождение проблем в этих системах позволит уменьшить недовольство B2B сектора и, возможно, решить проблемы B2C сектора. 

## Предлагаемое решение
Нужно будет добавить
- FluentBit для сбора логов с приложений 
- OpenSearch реализации для хранения и движка поиска по логам
- OpenSearch Dashboards для интерфейса доступа к логам

Для сбора логов из приложений и передачи их в OpenSearch нужно будет настроить пайплайны FluentBit. Для хранения и обработки логов потребуется настроить индексы OpenSearch. Для просмотра логов нужно будет настроить дешборды в OpenSearch Dashboards.

В логах не должны присутствовать чувствительные данные. Это можно обеспечивать на уровне кода и дополнительно фильтровать/маскировать данные в FluentBit. Доступ к логам должны иметь администраторы и разработчики системы.

Для каждого сервиса должен быть свой индекс темплейт для поддержки независимых маппингов в гетерогенной среде с уже устоявшимися сервисами. ISM-политики: для горячих индексов требуется настроить ролловер политики с сроком 1 день, перенос на холодное хранение с ролл-апом через неделю и удаление по истечении 60 дней. 

## Алёртинг, анализ
- Оповещения при возникновении ERROR логов
- Оповещения при обнаружении отсутствия обновления статуса заказа дольше порогового значения для определения зависших заказы
- Оповещения при увеличении количества создания заказов выше порогового значения: например 100 в минуту или кратно выше среднего за скользящее окно времени
