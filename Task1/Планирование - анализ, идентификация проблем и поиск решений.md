# Планирование: анализ, идентификация проблем и поиск решений

## Проблемные места
### Существующие проблемные места:
- Длительный рассчёт стоимости производства (2-30 мин)
- Потери заказов и невозможность отследить место утери данных
- Длительный цикл тестирования и исправления релиза
- Длительное получение списка заказов операторами MES
- Неконтролируемые нагрузки на систему ввиду прямого доступа к API рассчётов для внешних пользователей 

### Потенциальные проблемные места:
- Ограниченность масштабирования системы
- Множественные единые точки отказа

## Инициативы
- Разделить функционала рассчёта стоимости производства и ведения списка заказов
- Создать слой кэширования актуальных списка заказов для операторов MES
- Создать автоматизированные наборы тестов для проверки релиза
- Обеспечить трассировку и логирование запросов в системе и исправить утерю данных
- Разделить внутренний и внешний API рассчётов
- Обеспечить готовность системы к горизонтальному масштабированию и инструменты для этого

## Приоритеты

Первый квартал
1. **Обеспечить трассировку и логирование запросов в системе и исправить утерю данных**. Потеря заказов в данный момент является той самой важной и срочной проблемой, которую надо решить в первую очередь. Для принятия решения о конкретных исправлениях в системе требуется система трассировки запросов, которая позволит определить конкретные сценарии и места потери данных.
2. **Создать слой кэширования актуальных списка заказов для операторов MES**. Задержки в работе операторов MES приводят к снижению их эффективности, что напрямую влияет на доходы и предприятия и на лояльность сотрудников.
3. **Разделить внутренний и внешний API**. Предоставление внутреннего API рассчётов для пользования внешним потребителям ведёт невозможности контроля нагрузки на этот жизненно важный компонент, что может повлечь за собой перебои в работе всего бизнеса. Разделение API позволит реализовать паттерны Rate Limit, Backpressure, Circuit Breaker для защиты компонента. 

Второй квартал
4. **Создать автоматизированные наборы тестов для проверки релиза**. Отсутствие автоматизированных наборов тестов не позволяет проводить непрерывное тестирование релизов во время разработки и приводит к длительным циклам регресса. Это в свою очередь снижает гибкость и конкурентность бизнеса.
5. **Разделить функционала рассчёта стоимости производства и ведения списка заказов**. Функционал рассчёта является центральным механизмом, используемым всеми частями системы. Разделение функционала позволит разделить нагрузку на список заказов и на рассчёт стоимости и позволит горизонтально масштабировать функцию рассчёта стоимости.

Далее
6. **Обеспечить готовность системы к горизонтальному масштабированию и инструменты для этого**. Система справляется с текущми нагрузками, но нагрузки растут. Кроме того, вся инфрастуктура существует в едином экземпляре и ошибка в одном сервисе приводит к перебоям работы. В дальнейшем потребуется справляться с увеличенной нагрузкой и отказами отдельных сервисов.
